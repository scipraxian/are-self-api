# Fixture Separation — Claude Code Implementation Prompt

Read CLAUDE.md, STYLE_GUIDE.md, and this entire document before making any changes.
This is a blocking ticket. No other work should run concurrently.

## What You're Doing

Splitting every app's monolithic `initial_data.json` into up to three purpose-specific
fixture files, migrating all UUID-keyed fixture records to deterministic UUID5 values, and
updating all infrastructure (test base classes, install script, `core_dump` management
command, documentation) to match.

## Why

1. **Tests are too slow.** `CommonFixturesAPITestCase` loads ~270KB / 11,567 lines of fixture
   data for every test class. Most of that is starter-kit instance data that tests don't need.
2. **Shipping should be configurable.** Reference vocabulary (statuses, types, effector
   definitions) must always load. Starter-kit instances (sample identities, environments,
   pathways) should be optional.
3. **UUID PKs lack a stable contract.** Integer PK models have class-level constants
   (`Effector.FRONTAL_LOBE = 8`). UUID-keyed models in fixtures use random UUID4 values dumped
   from the dev database — they can't serve as stable references. Deterministic UUID5 values
   fix this.

## The Three-File-Per-App Structure

Each app that has fixtures gets up to three files in its `fixtures/` directory:

### `reference_data.json`
- **Contains:** Integer-PK lookup/vocabulary tables. Statuses, types, modes, capabilities,
  effector definitions, tool definitions, addon phases, shift types, etc.
- **Loaded by:** Install script (always), tests (always), Docker (always).
- **Rule:** No UUID-keyed records. No instance data. Pure vocabulary.

### `starter_kit.json`
- **Contains:** UUID5-keyed instance records that make the system functional out of the box.
  Sample identities, identity discs, project environments, neural pathways, neurons, axons,
  AI models, AI model providers, iteration definitions with shift definitions, context
  variables, etc.
- **Loaded by:** Install script (always — the 10-year-old needs this), Docker (always).
  Tests do NOT load this.
- **Rule:** All UUID PKs must be UUID5 with the project namespace (see below). May reference
  integer PKs from `reference_data.json` in the same or other apps.

### `test_data.json`
- **Contains:** UUID5-keyed instance records (test namespace) purpose-built for the test
  suite. Minimal — just enough to satisfy FK constraints for tests that need instance data.
- **Loaded by:** Tests only (via `CommonFixturesAPITestCase`).
- **Rule:** All UUID PKs must be UUID5 with the test namespace. Must NOT duplicate or depend
  on `starter_kit.json`. Self-contained.

**Not every app needs all three files.** Apps with only integer-PK reference data (like
`frontal_lobe`, `prefrontal_cortex`) will only have `reference_data.json`.

## UUID5 Namespace Design

### The Pattern (already proven in Hypothalamus)

The `ollama_fixture_generator.py` already uses:
```python
uuid.uuid5(uuid.NAMESPACE_DNS, f'areself.ollama.{model_slug}')
```

Generalize this into a project-wide convention.

### Namespaces

Define in `common/constants.py`:
```python
import uuid

# Deterministic UUID5 namespaces for fixture data.
# Project namespace: shipping fixtures (starter_kit.json).
# Test namespace: test-only fixtures (test_data.json).
# Runtime records use UUID4 (the UUIDIdMixin default) and never collide.
FIXTURE_NS = uuid.uuid5(uuid.NAMESPACE_DNS, 'are-self.com.fixtures')
TEST_FIXTURE_NS = uuid.uuid5(uuid.NAMESPACE_DNS, 'are-self.com.test-fixtures')
```

### UUID5 Generation Convention

```python
# Starter kit records:
uuid.uuid5(FIXTURE_NS, 'identity.identity.pm-blueprint')
uuid.uuid5(FIXTURE_NS, 'identity.identitydisc.steve-worker')
uuid.uuid5(FIXTURE_NS, 'environments.projectenvironment.are-self-default')
uuid.uuid5(FIXTURE_NS, 'central_nervous_system.neuralpathway.canonical-tick')

# Test records:
uuid.uuid5(TEST_FIXTURE_NS, 'identity.identity.test-identity-1')
uuid.uuid5(TEST_FIXTURE_NS, 'identity.identitydisc.test-disc-1')
```

The slug portion should be human-readable and describe the record's purpose.

### Hypothalamus AIModel — Keep Existing Convention

The 4 AIModel records already use `uuid5(NAMESPACE_DNS, f'areself.ollama.{slug}')`.
**Do not change these.** They were generated by the fixture generator and downstream tools
may depend on the values. Document the divergence: AIModel uses its own namespace convention
because models are generated programmatically, not hand-authored.

### Constants on Model Classes (Optional Enhancement)

**UUID constants already exist on model classes and are used in production code:**

```python
# identity/models.py
class Identity(...):
    THALAMUS = UUID('14148e25-283d-4547-a17d-e28d021eba07')  # used nowhere directly yet

class IdentityDisc(...):
    THALAMUS = UUID('15ca85b8-59a9-4cb6-9fd8-bfd2be47b838')  # used in thalamus/api.py:106

# environments/models.py
class ProjectEnvironment(...):
    DEFAULT_ENVIRONMENT = uuid.UUID('44b23b94-6aae-4205-ae67-2f8c021c67aa')  # used in central_nervous_system/utils.py:86
```

**These constants MUST be updated to the new UUID5 values.** They are actively referenced:
- `IdentityDisc.THALAMUS` → `thalamus/api.py` line 106
- `ProjectEnvironment.DEFAULT_ENVIRONMENT` → `central_nervous_system/utils.py` line 86

When you generate the new UUID5 values, update both the fixture files AND these class
constants simultaneously. The constant names stay the same — only the UUID values change.

## Fixture Load Order

### Install script (`are-self-install.bat`)

Replace the single `loaddata initial_data.json` with:

```batch
echo [7/9] Loading reference data...
.\venv\Scripts\python.exe manage.py loaddata reference_data.json

echo [8/9] Loading starter kit...
.\venv\Scripts\python.exe manage.py loaddata starter_kit.json
```

Django discovers `reference_data.json` across all apps via INSTALLED_APPS order, then
`starter_kit.json` the same way. Reference data has no UUID FK dependencies, so load order
within that pass doesn't matter. Starter kit may have cross-app UUID FK dependencies —
Django's `loaddata` defers FK constraint checks within a single command (it wraps in a
transaction), so a single `loaddata starter_kit.json` that picks up all apps should work.

**If cross-app FK constraints fail in a single pass**, fall back to explicit ordered loading:
```batch
.\venv\Scripts\python.exe manage.py loaddata ^
    hypothalamus/fixtures/starter_kit.json ^
    identity/fixtures/starter_kit.json ^
    environments/fixtures/starter_kit.json ^
    central_nervous_system/fixtures/starter_kit.json ^
    temporal_lobe/fixtures/starter_kit.json
```

### Test base classes (`common/tests/common_test_case.py`)

```python
class CommonTestCase(APITestCase):
    """A test case with reference data only."""
    fixtures = ('reference_data.json',)
    # ... (setUp unchanged)


class CommonFixturesAPITestCase(CommonTestCase):
    """A test case with reference data + test-specific instance data."""
    fixtures = [
        'parietal_lobe/fixtures/reference_data.json',
        'hypothalamus/fixtures/reference_data.json',
        'identity/fixtures/reference_data.json',
        'temporal_lobe/fixtures/reference_data.json',
        'prefrontal_cortex/fixtures/reference_data.json',
        'environments/fixtures/reference_data.json',
        'peripheral_nervous_system/fixtures/reference_data.json',
        'central_nervous_system/fixtures/reference_data.json',
        'frontal_lobe/fixtures/reference_data.json',
        # Test instance data
        'hypothalamus/fixtures/test_data.json',
        'identity/fixtures/test_data.json',
        'environments/fixtures/test_data.json',
        'peripheral_nervous_system/fixtures/test_data.json',
        'central_nervous_system/fixtures/test_data.json',
    ]
```

**Important:** `CommonTestCase.fixtures = ('reference_data.json',)` uses the auto-discovery
shorthand (no path prefix). This works because each app's `fixtures/` dir is on Django's
fixture search path. It will find ALL `reference_data.json` files across all apps. Verify
this works — if not, use explicit paths.

**Actually — potential issue:** The tuple `('reference_data.json',)` might load only ONE
file named `reference_data.json` (the first one Django finds), not all of them. The current
`('initial_data.json',)` works because `initial_data` is a Django magic name that
auto-loads from all apps. `reference_data.json` is NOT magic.

**Solution:** Either (a) keep the base class using explicit paths for all reference_data
files, or (b) name it `initial_data.json` (but that defeats the purpose of the rename).
Recommend option (a): list all reference_data paths explicitly in `CommonTestCase`.

## Updating `core_dump` Management Command

The `core_dump` command currently dumps everything non-transactional into one file. It needs
to be split-aware:

1. For each app, classify models into reference (integer PK) vs instance (UUID PK).
2. Dump reference models to `reference_data.json`.
3. Dump instance models to `starter_kit.json`.
4. Leave `test_data.json` hand-maintained (not generated by core_dump).

The classification can be done by checking whether the model's PK field is a UUIDField:
```python
from django.db.models import UUIDField

for model in app_config.get_models():
    pk_field = model._meta.pk
    if isinstance(pk_field, UUIDField):
        instance_models.append(...)
    else:
        reference_models.append(...)
```

## Hazards to Watch For

1. **`initial_data.json` is a Django magic name.** `loaddata initial_data` auto-discovers
   across all apps. `reference_data.json` does not get this treatment. The test base classes
   MUST use explicit paths.

2. **FK constraint ordering.** `loaddata` defers constraints within a single command. If you
   load `reference_data.json` as a single command, all apps' reference data loads in one
   transaction. Same for `starter_kit.json`. But if you load them as separate per-app
   commands, FK ordering matters.

3. **Cross-app UUID FK references.** The CNS `starter_kit.json` references identity disc
   UUIDs, environment UUIDs, and hypothalamus integer PKs. All referenced records must exist
   before the CNS starter kit loads — or be in the same `loaddata` transaction.

4. **The `core_dump` command uses `dumpdata`.** `dumpdata` serializes whatever is in the
   database. If the database has UUID4 values (from before the migration), the dump will
   contain UUID4 values. After the migration, ensure the live database is updated before
   running `core_dump`. Or: apply a data migration that converts existing UUID4 PKs to UUID5.

5. **`CommonTestCase` currently uses `('initial_data.json',)` — a single-element tuple.**
   This relies on Django's magic auto-discovery for `initial_data`. The replacement must be
   a list of explicit paths.

6. **M2M fields in fixtures.** Some fixture records have M2M fields (like `aimodel.roles`,
   `aimodel.capabilities`). These reference integer PKs in other tables. Ensure M2M target
   tables are loaded before the records that reference them (they should be, since M2M
   targets are reference data).

7. **The `test_agents.json` file.** Currently loaded by `CommonFixturesAPITestCase` as a
   separate entry. Replace with `peripheral_nervous_system/fixtures/test_data.json`.

8. **Existing tests may hardcode UUID4 values.** Search the test suite for any hardcoded
   UUIDs that match the old fixture values. Update them to reference the new UUID5 values.
   ```bash
   grep -rn "14148e25\|59189dba\|8a1b03d5\|0db0d16e\|108f68c7\|15ca85b8\|44b23b94\|4d82fab0\|c1b7a124" --include="*.py" .
   ```

9. **Model class UUID constants.** Three constants on model classes reference old UUID4
   fixture values and are used in production code:
   - `Identity.THALAMUS` in `identity/models.py`
   - `IdentityDisc.THALAMUS` in `identity/models.py` → referenced by `thalamus/api.py:106`
   - `ProjectEnvironment.DEFAULT_ENVIRONMENT` in `environments/models.py` → referenced by
     `central_nervous_system/utils.py:86`
   These MUST be updated to the new UUID5 values or the system will break at runtime.

10. **Fixture files also cross-reference UUID4 values.** The CNS `initial_data.json`
    contains `neuroncontext` records whose values point to `IdentityDisc` UUID4 PKs, and
    `neuralpathway` records whose `environment` field points to `ProjectEnvironment` UUID4
    PKs. Every cross-file UUID reference must be updated consistently.
